# 내 담당(유저 · 인증 · 전시) 작업 정리

> 출처: [기존 구현 vs 와이어프레임 차이 체크리스트](기존%20구현%20vs%20와이어프레임%20차이%20체크리스트.md)에서 **유저·인증·전시** 도메인 항목만 추림.

## 한눈에 보기

| 도메인 | 수정(🔧) | 신규(🆕) | 요약 |
|---|---|---|---|
| 인증 auth | 0 | 0 | **변경 없음** — 로그인/재발급/로그아웃 그대로 |
| 유저 user | 1 | 2 | stats 의미 변경 / 알림설정·회원탈퇴 신규 |
| 전시 exhibition | 7 | 2 | 목록·상세·등록 대거 확장 / 배너·전시관검색 신규 |
| (관심 전시 북마크) | — | 3 | 신규 도메인, **전시·유저 경로에 얹힘** → 사실상 내 작업 |

전체 개수: 수정 8건 + 신규 7건.

# 1. 인증 (auth) — ✅ 변경 없음

소셜 로그인 · 게스트 로그인 · 토큰 재발급 · 로그아웃 모두 기존 구현 그대로 사용한다. **추가·수정 작업 없음.**

# 2. 유저 (user)

## 🔧 수정

### U-1. `GET /users/me`의 `stats.bookmarkCount` 의미 변경  · P0
- **현재**: 북마크한 **기록** 수 (`UserDto.MeResponse.Stats`)
- **바꿀 것**: 프로필 "관심 전시 10"에 맞춰 **전시 북마크 수**로 교체
- **주의**: 전시 북마크(아래 B-북마크) 도입이 선행돼야 함 → 세트로 진행
- **파일**: `UserDto.MeResponse.Stats`, 프로필 조회 Facade

## 🆕 신규

### U-2. 알림 설정 조회/수정  · P1
- **API**: `GET /api/v1/users/me/notification-settings`, `PUT …`
- **바디**: `{ "remindEnabled": true, "noticeEnabled": false }`
- **용도**: 프로필 > 설정의 리마인드 알림 / 공지 알림 토글
- **참고**: 저장 대상 필드를 User 엔티티에 추가할지, 별도 설정 엔티티로 뺄지 결정

### U-3. 회원 탈퇴  · P1
- **API**: `DELETE /api/v1/users/me` (바디 없음, 응답 `data:null`)
- **처리**: soft-delete + 토큰 무효화
- **참고**: 설정 > 계정 > 회원 탈퇴

> 관심 전시 목록(`GET /users/me/bookmarks`)도 유저 경로에 있지만, 관심 전시(북마크) 신규 섹션(4장)에 함께 정리했다.

# 3. 전시 (exhibition)

와이어프레임에서 가장 많이 손대는 도메인. **목록·상세·직접등록 세 API 확장 + 배너·전시관검색 신규.**

## 🔧 수정 — 전시 목록 `GET /api/v1/exhibitions`

한 엔드포인트에 여러 변경이 겹친다. 아래를 함께 설계하는 게 효율적.

### E-1. 응답 필드 추가 (목록·상세 공통)  · P0
`ListItemResponse`·`DetailResponse`에 아래 필드 신설:

| 필드 | 의미 | 비고 |
|---|---|---|
| `artistSummary` | "김미경 외 10인" 표기 | 목록엔 아예 없음(상세 `artists`만 존재) |
| `dDay` | 종료 D-n (홈 D-5 배지) | 종료 후 null |
| `free` | 무료 여부 | `price` 문자열에서 파생(판별 규칙 확정 필요 → C-6) |
| `bookmarked` | 로그인 시 관심 여부 | 비로그인 false |
| `recorded` (상세만) | 내 기록 존재 여부 | "기록하기" 버튼 분기 |

### E-2. 다중 필터  · P1
- **현재**: `region`·`category`가 단일 문자열
- **바꿀 것**: 콤마 다중값(`region=SEOUL,GYEONGGI`) 파싱 → 필터 바텀시트 칩 다중 선택 대응

### E-3. 홈 섹션 조건 + `section` 파라미터  · P0
- **추가**: `section=ending-soon | opening-this-month | free` (+ `period=month|week`)
- **조건**: 곧 종료 = 종료일 ≤ 오늘+7일 / 이번 달·주 = 시작일 ∈ 당월·당주 / 무료 = 관람료 0원·무료
- **활용**: 홈 미리보기는 `size=2`, 전체보기는 `size=20`(같은 API)

### E-4. 커서 페이지네이션 전환  · P0 ⚠️ 모든 목록의 기반
- **현재**: 오프셋(`page=0&size=20`) + `PageResponse`
- **바꿀 것**: `cursor`+`size` → `CursorResponse{content, nextCursor, hasNext, totalCount}` (QueryDSL no-offset, `totalCount`는 별도 count)
- **주의**: CLAUDE.md 페이징 컨벤션 문구도 갱신 필요. 정렬 타이브레이커(E-6)와 커서 키를 함께 설계

### E-5. 거리순 정렬  · P2
- **추가**: `sort=distance` + `lat`/`lng`(프론트가 항상 제공, 없으면 400)
- **처리**: 기존 `gpsX/gpsY`로 DB 거리 계산 정렬. 서버 fallback 없음

### E-6. 정렬 타이브레이커  · P1
- **현재**: 단일 키 정렬(동률 시 순서 비결정)
- **바꿀 것**: 동률이면 이름 가나다순, **첫 글자 한글이 영문보다 우선**
- **주의**: MySQL collation 기본이 아니라 정렬 표현식(한글 우선 파생 키) 필요 → 커서 키에도 포함(E-4 연계)

### E-7. 검색어 최소 2글자 검증  · P1
- **바꿀 것**: 1글자 keyword → 400 `INVALID_INPUT`, 비교 시 공백 제거·대소문자 무시 정규화

## 🔧 수정 — 전시 직접 추가 `POST /api/v1/exhibitions/custom`

### E-8. 등록 입력 항목 확장  · P1
`CustomCreateRequest`에 아래 추가:

| 필드 | 의미 |
|---|---|
| `exhibitionForm` | 개인전/단체전/기획전/아트페어 (`SOLO|GROUP|CURATED|ART_FAIR`) |
| `artistName` | 개인전일 때 필수 → "개인전 · 김선명" 표기 |
| `venueId` | 전시관 검색(E-10) 선택 시. 없으면 `place` 직접 입력 |

### E-9. 장르(category) 코드 확장  · P1
- **현재**: `PAINTING, PHOTO, MEDIA, SCULPTURE, ETC` (5종)
- **바꿀 것**: `DESIGN, CRAFT, ARCHITECTURE, PERFORMANCE` 추가 → 9종
- **주의**: 기존 데이터 ETC→재분류 여부 확인

## 🆕 신규 — 전시

### E-10. 홈 배너 조회  · P0
- **API**: `GET /api/v1/exhibitions/banners`
- **응답**: 캐러셀 최대 3개 배열. 운영자 지정 우선 → 없으면 진행 중 전시 조회수 상위
- **주의**: 홈은 `/home` 단일 집계가 아니라 **배너 1콜 + 섹션 3콜(E-3) = 병렬 4콜**

### E-11. 전시관 검색(자동완성)  · P1
- **API**: `GET /api/v1/venues?keyword=`
- **응답**: `{venues:[{venueId, name, address, region}]}` 상위 20개
- **선행**: 전시관 마스터 테이블 필요. E-8의 `venueId` 연동

# 4. 관심 전시 (북마크) — 🆕 신규 도메인, 내 경로에 얹힘

기존 "기록 북마크"와 별개인 **전시 북마크**. 엔드포인트가 `/exhibitions`·`/users` 경로라 사실상 내 작업이다.

### B-1. 전시 북마크 등록/해제  · P0
- **API**: `POST` / `DELETE /api/v1/exhibitions/{exhibitionId}/bookmark` (멱등)
- **응답**: `{ "exhibitionId": 51, "bookmarked": true }`
- **연계**: E-1의 `bookmarked` 필드, U-1의 `bookmarkCount`가 이 데이터에 의존

### B-2. 관심 전시 목록  · P0
- **API**: `GET /api/v1/users/me/bookmarks?sort=latest|ending`
- **응답**: `CursorResponse<ExhibitionListItem>` (커서, 프로필 "관심 전시" 화면)

---

# 5. 파일 업로드 연계 (참고)

전시 포스터 등록(E-8)은 `POST /api/v1/files`(신규, 공통 도메인)로 URL을 먼저 받아야 한다. 이 API 자체는 공통/support 담당이지만, 전시 등록 플로우가 의존하므로 순서를 맞춰 진행.

# 7. 진행 전 확정받을 것 (내 도메인 관련)

- **C-6 무료(free) 판별 규칙**: CATALOG price 문자열 → 무료 파생 기준("무료"/"0원" 포함?) — E-1 `free`에 필요
- **C-3 tasteKeywords 산출 기준**: 프로필 "나의 감정 키워드" 집계 로직이 현재 구현과 일치하는지 (유저 도메인)
- **ageGroup 정밀도**: enum이 10년 단위(`TWENTIES`)인데 와이어프레임 프로필 수정은 "20~24세" 5년 단위 → enum 재정의 필요 여부 (유저 도메인)
