# 전시 방문 기록(다녀온 전시) — 전시 정보 스냅샷 설계

- 작성일: 2026-07-02
- 브랜치: `14-49-백엔드-전시-API-리팩토링`
- 상태: 설계 승인됨 → 구현 계획 대기. **아래 §0 갱신 필독**(A안 복귀·엔드포인트 경로 변경)
- 관련: `docs/04_전시_구현.md`(전시 도메인), `V1__create_record_tables.sql`(기록 도메인), 스펙 ②(전시 목록·상세·필터)

## 0. 갱신 (2026-07-02, A안 확정 반영)

이 스펙은 B안(실시간 프록시) 전제로 작성됐으나, 이후 **A안(배치 시딩 + DB 조회)** 으로 확정됐다(스펙 ② §11). 그에 따른 변경:
- **CATALOG가 DB에 상주**(안정적 Long id 복귀) → 스냅샷 파생 id 불필요. 기록은 전시의 실제 DB id 참조.
- 기록 생성 시 전시 조회(`ExhibitionFacade.getDetail`)는 **DB 기반**(외부 실시간 호출 아님) → 빠르고 외부 장애 무관.
- **스냅샷은 그대로 유지**. 이유가 "휘발성 방지"에서 → "**시각 고정(그 순간의 전시 정보) + 매시 재동기화 덮어쓰기·향후 정리로부터 보호**"로 이동.
- **스냅샷 필드**: `title·type·posterUrl·place·region·startDate·endDate`(작가는 소스 미제공이라 제외).
- **방문목록 엔드포인트 경로**: 기존 `GET /api/v1/records` 재사용 대신 **`GET /api/v1/records/exhibitions/visited`** 로 노출(사용자 요청). 소유는 기록 도메인 유지(전시 도메인과 양방향 결합 회피).
- 본문 §4·§8의 "실시간 fetch" 표현은 위 A안 기준으로 읽을 것.

## 1. 배경 · 문제

전시 도메인은 방금 **B안 리팩토링**으로 CATALOG(공공데이터 공식 전시)를 DB에 적재하지 않고
**매 조회마다 외부 API를 실시간 호출**하는 프록시로 바뀌었다. CATALOG의 식별자는 DB PK가 아니라
원천 seq 파생값이며, **그 전시가 실시간 API 응답 window에 남아 있을 때만** 제목·포스터를 되살릴 수 있다.

한편 "내가 다녀온 전시"는 이미 구현된 `records`(전시 관람 기록) 도메인이 담당한다
(작성/목록/상세/수정/삭제/북마크, 관람일·감상·감정·키워드·미디어·AI요약 보유). 이 기록은 전시를
`exhibition_id`(Long) **참조로만** 들고 있고, 전시 제목·포스터 등 표시정보는 저장하지 않는다.

두 성질이 충돌한다:
- 기록은 **영구적**(과거에 다녀온 전시).
- CATALOG는 **실시간·휘발성**(진행 종료되면 피드에서 빠짐).

→ 과거에 다녀온 전시가 API 피드에서 사라지면, 기록은 `exhibition_id`만 남고 **전시 제목·포스터를
불러올 수 없는 "고아 기록"** 이 된다.

## 2. 목표 · 비목표

**목표**
- "내가 다녀온 전시 목록"(별도 조회 기능, `GET /api/v1/records`)이 전시가 실시간 피드에서 사라져도
  **전시 제목·포스터·장소 등을 항상 표시**할 수 있게 한다.

**비목표**
- 실시간 전시 목록(`GET /api/v1/exhibitions`)에 "다녀온" 표시를 섞는 것. (별도 조회로 유지)
- `records` 도메인의 기존 컨벤션 불일치(성공 201 반환, `RecordService` 네이밍, X-User-Id 흔적) 정리. (범위 밖)
- 기록의 감상·감정·키워드·미디어·AI요약 등 기존 필드/로직 변경.

## 3. 결정 요약

| 항목 | 결정 | 근거 |
|---|---|---|
| 표시 기준 | 전시 **제목·공식 포스터 중심** | "어떤 전시였는지"가 다녀온 목록의 본질 |
| 스냅샷 출처 | **서버가 재조회해 박제(B안)** | 위조 불가·단일 진실원천. 클라 신뢰 대신 서버 권위 |
| 미해결 시 | **기록 작성 실패(가안)** | 식별 불가한 전시는 권위 스냅샷 불가 → 막는다. 단순·일관 |
| 스냅샷 시점 | **생성 시 1회 확정, 수정 시 불변** | 기록=그 순간의 박제. 수정 요청엔 전시 정보가 없음 |

## 4. 아키텍처 · 데이터 흐름

```
POST /api/v1/records  (exhibitionId + 감상/감정/미디어 …)
  └─ RecordService.create(userId, request)
       ├─ ExhibitionFacade.getDetail( ExhibitionCriteria.Detail(exhibitionId, userId) )
       │     · CATALOG → 외부 API 실시간 조회 / CUSTOM → DB
       │     · 접근제어 재사용: CATALOG 공개, CUSTOM 본인만(타인 403)
       │     · 미해결 → EXHIBITION_NOT_FOUND(404) 전파 → 기록 생성 실패
       ├─ ExhibitionResult.Detail → ExhibitionSnapshot(VO)로 추출
       └─ Record.create(userId, exhibitionId, snapshot, …) 저장

GET /api/v1/records, GET /api/v1/records/{id}
  └─ 저장된 스냅샷을 그대로 반환 (외부 재호출 없음)
```

**의존 방향**: `application.record`(RecordService) → `application.exhibition`(ExhibitionFacade).
단방향, 순환 없음. 기록 도메인이 전시 도메인의 유스케이스(조회+접근제어)를 재사용한다.

## 5. 스키마 (V7 마이그레이션)

`records` 테이블에 스냅샷 컬럼 추가. `exhibition_id`는 참조/필터용으로 유지(재조회 시 활용 가능).

```sql
-- V7__add_record_exhibition_snapshot.sql
alter table records
    add column exhibition_title      varchar(100)  not null,
    add column exhibition_type       varchar(20)   not null,   -- CATALOG / CUSTOM
    add column exhibition_poster_url varchar(2048) null,
    add column exhibition_place      varchar(200)  null,
    add column exhibition_region     varchar(20)   null,
    add column exhibition_category   varchar(20)   null,
    add column exhibition_start_date date          null,
    add column exhibition_end_date   date          null;
```

> 주의: `not null` 컬럼을 추가하므로 기존 데이터가 있으면 마이그레이션이 실패한다. 현재 `records`는
> 운영 데이터가 없는 개발 단계(데모/로컬)라는 전제. 데이터가 있다면 default/backfill 단계를 추가한다.

## 6. 도메인 변경 (`domain/record`)

**`ExhibitionSnapshot`(신규 VO, record)** — 전시 표시정보 묶음. 영속화는 Entity 원시 컬럼으로.
```
record ExhibitionSnapshot(
    String title,            // 필수(비면 검증 실패)
    String type,             // 필수. "CATALOG" / "CUSTOM"
    String posterUrl, String place,
    String region, String category,   // enum 이름 문자열
    LocalDate startDate, LocalDate endDate)
```
- `type`/`region`/`category`는 **문자열**로 둔다 — `ExhibitionResult.Detail`이 이미 enum 이름(String)을 내려주므로
  재파싱이 불필요하고, 기록 도메인이 전시 enum에 의존하지 않아 결합도가 낮다. DB 컬럼도 varchar라 그대로 맞물린다.

**`Record` 엔티티**
- 스냅샷 원시 컬럼(§5) 추가. `@Getter`, Setter 없음(기존과 동일).
- `create(...)` 시그니처에 `ExhibitionSnapshot snapshot` 추가(생성자에서 언팩·저장, `title` 공백 검증).
- 스냅샷은 생성 후 불변 — 기존 `replaceContent(...)`(수정)은 스냅샷을 건드리지 않는다.

## 7. 애플리케이션 변경 (`application/record`)

**`RecordService`**
- `ExhibitionFacade` 주입 추가(`@RequiredArgsConstructor` + `private final`).
- `create(...)`: `exhibitionFacade.getDetail(new ExhibitionCriteria.Detail(request.exhibitionId(), userId))`로
  전시를 조회 → `ExhibitionResult.Detail`을 `ExhibitionSnapshot`으로 변환 → `Record.create(..., snapshot, ...)`.
- 미해결/접근불가는 `getDetail`이 던지는 `EXHIBITION_NOT_FOUND`(404)/`FORBIDDEN`(403)을 그대로 전파.
- `update/get/search/delete/bookmark`는 스냅샷을 읽기만 하거나 무관 — 로직 변경 없음(응답 매핑만 확장).

## 8. 인터페이스 변경 (`interfaces/record`)

- **요청**: `RecordCreateRequest` 변경 없음(여전히 `exhibitionId` + 기록 필드). 서버가 스냅샷을 채운다.
- **응답 확장**:
  - `RecordListItemResponse`에 `exhibitionTitle`, `exhibitionPosterUrl`, `exhibitionPlace`,
    `exhibitionRegion`, `exhibitionCategory`, `exhibitionStartDate`, `exhibitionEndDate` 추가.
  - `RecordDetailResponse`에 동일 스냅샷 필드 추가.
  - `RecordCreateResponse`는 **변경하지 않는다**(생성 결과 식별자 위주 유지).

## 9. 에러 처리

| 상황 | 코드 | HTTP |
|---|---|---|
| 전시 미해결(피드에서 사라짐/오타) | `EXHIBITION_NOT_FOUND` | 404 |
| 타인의 CUSTOM 전시로 기록 시도 | `FORBIDDEN` | 403 |
| 감정 미선택·관람일 미래·미디어 규칙 위반 | 기존 `RECORD_*` | 400 |

신규 에러 코드 불필요 — 전시 도메인 코드를 그대로 전파.

## 10. 테스트

- **Record 엔티티(단위)**: `create`가 스냅샷을 저장하는지, `title` 공백이면 실패, 수정이 스냅샷을 안 바꾸는지.
- **RecordService(단위, Mockito)**: `ExhibitionFacade.getDetail` 목 →
  - 성공: 반환 Detail이 스냅샷으로 기록에 저장.
  - 전시 미해결(getDetail이 404 throw): 기록 생성 실패, 저장 호출 안 됨.
  - 타인 CUSTOM(403 throw): 실패 전파.
- **컨트롤러**: 목록/상세 응답에 스냅샷 필드 포함.
- **통합(end-to-end) 1케이스**: 로그인 → (CATALOG 목 세팅) 전시 조회 → 기록 작성 → `GET /records`에
  전시 제목·포스터가 스냅샷으로 나오는지. CATALOG 목을 빈 목록으로 바꿔도(=피드에서 사라짐)
  다녀온 목록엔 제목·포스터가 그대로 남는지 확인.

## 11. 리스크 · 열린 점

- **(수용됨) 방금 다녀온 전시가 API window 밖이면 기록 실패**: B안+가안의 대가. 부담되면 C안(클라값 폴백)으로 후속 완화 가능.
- **not null 컬럼 추가 마이그레이션**: 기존 records 데이터가 없다는 전제(§5).
- **application→application 의존**(record→exhibition): 순환 없음. 커지면 좁은 조회 포트로 분리 고려.
