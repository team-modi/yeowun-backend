# 요구사항 정리

---

## 1. 목적 및 범위

### 목적

사용자가 전시를 탐색하고, 다녀온 전시의 감상을 기록하고, 시간이 지난 뒤 그 기록을 다시 꺼내어 회고(리마인드)할 수 있는 전시 아카이빙 서비스를 구축한다. 전시 데이터는 외부 공공데이터 API에서 자동 수집하고, AI가 감상문 작성과 감정 변화 요약을 돕는다.

### 범위

본 문서가 다루는 범위는 다음 아홉 가지 도메인으로 **한정**한다.

- **인증(Auth)** — 소셜(카카오/구글) 로그인과 게스트 로그인. 자체 JWT(access/refresh)를 발급한다.
- **사용자(User)** — 프로필, 알림 설정, 회원 탈퇴. 소셜 연결(SocialAccount)을 보유한다.
- **전시(Exhibition)** — 탐색의 대상. 외부 API 수집 전시(CATALOG)와 사용자 직접 등록 전시(CUSTOM)를 하나의 모델로 다룬다.
- **전시관(Venue)** — 개인 전시 등록 시 전시관명 자동완성에 쓰이는 마스터 데이터.
- **관심 전시(Bookmark)** — 사용자가 전시를 관심 등록하는 행위. 멱등 토글로 동작한다.
- **기록(Record)** — 다녀온 전시의 감상. 감정·미디어를 함께 남기고, 전시 정보는 스냅샷으로 박제한다.
- **AI 감상문(Record AI)** — '질문으로 작성' 플로우. 전시 맥락 기반 질문 생성과 답변 기반 감상문 다듬기.
- **리마인드(Remind)** — 7일 이상 지난 기록을 다시 소환해 "지금 다시 보니 어떤가"를 남기는 회고. AI가 감정 변화를 요약한다.
- **알림(Notification)** — 홈 종 아이콘의 알림 목록. 리마인드(REMIND)와 공지(NOTICE) 두 종류.

### 액터

- **비회원(Guest Visitor)** — 로그인 없이 전시 목록·배너·상세를 탐색한다.
- **회원(User)** — 소셜 또는 게스트 로그인한 사용자. 기록·리마인드·관심 전시·개인 전시 등록 등 모든 개인화 기능을 사용한다.
- **시스템(Scheduler)** — 부팅 시·매일 자정에 외부 전시 API를 동기화하고(신규만 적재), 동기화 직후 신규 적재분(미분류 행)의 장르를 분류하며, 상세 정보를 백필한다(분류기는 운영=AI, 개발=랜덤).

### 범위 고정

- 인증은 **FE 주도 소셜 로그인** 방식이다. FE가 provider 콜백에서 받은 code를 보내면 백엔드가 자체 JWT를 발급한다.
- 발급된 토큰은 **HttpOnly 쿠키**(access/refresh)로 내려가며, 본문 accessToken은 비쿠키 클라이언트 호환용으로 함께 반환한다.
- 게스트 로그인도 소셜 로그인과 **동일한 토큰 체계**를 사용하므로, 발급 즉시 로그인 전용 API를 그대로 쓸 수 있다.
- 성공 응답은 프로젝트 컨벤션에 따라 **전부 200**으로 통일한다(기록 작성 한 곳만 201). PATCH는 사용하지 않는다.

## 2. 유저 시나리오

### 2.1 회원 — 로그인

> 사용자는 카카오/구글 계정 또는 게스트로 로그인하여 서비스를 시작한다.

**시나리오**

1. 사용자가 소셜 로그인(카카오/구글)을 완료하면 FE가 인가 code를 백엔드로 보낸다.
2. 처음 로그인한 사용자는 자동으로 가입되고, 기존 사용자는 해당 계정으로 로그인된다.
3. 사용자가 소셜 계정 없이 게스트로 로그인한다.
4. access 토큰이 만료되면 refresh 토큰으로 재발급받는다.
5. 사용자가 로그아웃한다.

**드러나는 행위**

- 소셜 연결은 **(provider, providerUserId)** 로 유일하게 식별한다. 연결이 있으면 기존 User, 없으면 신규 User 생성 + 연결 저장.
- 신규 가입 시 소셜 동의항목(닉네임·이름·연령대·출생연도)을 프로필에 반영한다. **재로그인 시에는 이메일만 최신화**하고 사용자가 편집한 프로필을 덮지 않는다.
- FE가 보낸 redirectUri는 **화이트리스트로 검증**한다(신뢰하지 않음).
- refresh 재발급은 **회전(rotation)** 방식 — 재발급 시 새 refresh로 교체 저장한다.
- 게스트는 닉네임 "게스트", 프로필 미완 상태로 생성되며 이후 프로필 수정으로 채운다.
- 로그아웃은 **멱등** — refresh가 무효하거나 없어도 실패하지 않는다.

---

### 2.2 회원 — 프로필·설정

> 사용자는 자신의 프로필과 알림 설정을 관리하고, 원하면 탈퇴한다.

**시나리오**

1. 사용자가 내 프로필(닉네임·프로필 이미지·연령대·거주지역)과 취향 감정 키워드, 활동 통계를 조회한다.
2. 사용자가 프로필을 수정한다. 전달한 필드만 반영된다(부분 갱신).
3. 사용자가 알림 설정(리마인드·공지 수신 여부)을 조회하고 수정한다.
4. 사용자가 회원 탈퇴한다.

**드러나는 행위**

- 프로필 조회에는 **활동 통계**(기록 수·다녀온 전시 수·관심 전시 수)와 **감정 키워드**(내 기록의 감정 빈도순)가 함께 담긴다 — record·bookmark 도메인에서 실집계한다.
- 프로필 수정은 **PUT이지만 부분 갱신** — null이 아닌 필드만 반영하며, 갱신 완료 시 profileCompleted=true.
- 알림 설정은 **옵트아웃** 방식 — 가입 시 리마인드·공지 모두 기본 true.
- 탈퇴는 **soft-delete + refresh 토큰 무효화**. 탈퇴한 사용자는 조회되지 않는다.

---

### 2.3 (비)회원 — 전시 탐색

> 누구나 로그인 없이 전시를 탐색한다. 로그인하면 개인화(내 CUSTOM 전시·북마크 여부)가 반영된다.

**시나리오**

1. 사용자가 전시 목록을 조회한다. 검색어·섹션·지역·카테고리·날짜로 필터링하고, 다양한 기준으로 정렬한다.
2. 사용자가 홈 배너(진행 중 인기 전시 최대 3개)를 조회한다.
3. 사용자가 특정 전시의 상세 정보를 조회한다.

**드러나는 행위**

- 전시 탐색은 **로그인 없이** 가능하다(선택 인증). 비로그인은 CATALOG만, 로그인 시 본인의 CUSTOM 전시도 함께 노출된다.
- 목록은 **커서(키셋) 페이지네이션** — 기본 20건, 최대 50건.
- 정렬 기준: 최신순(latest, 기본), 종료 임박순(ending), 인기순(popular, 앱 내 조회수), 거리순(distance, lat/lng 필수).
- 섹션 필터: 곧 끝나는 전시(ending-soon, 종료일 오늘~+7일), 이번 달/주 시작(opening-this-month), 무료(free).
- **아무 필터가 없으면 오늘 진행 중인 전시**를 기본 노출한다.
- 검색어는 **최소 2글자**(전시명·전시장명 부분 일치).
- 상세 진입 시 조회수가 1 증가하고, CATALOG 전시는 최초 진입 때 외부 API에서 **상세 정보를 1회 지연 수집**해 캐시한다(실패 시 기본 필드만 반환, 다음 조회에서 재시도).
- 상세의 bookmarked(관심 여부)·recorded(기록 여부)는 요청자 기준으로 채운다(비로그인 false).

---

### 2.4 회원 — 개인 전시 등록

> 사용자는 목록에 없는 전시를 직접 등록해 기록의 대상으로 삼는다.

**시나리오**

1. 사용자가 전시관명을 입력하면 자동완성 목록(상위 20개)이 뜬다.
2. 사용자가 제목·기간·지역·카테고리·형태·작가·포스터를 입력해 개인 전시를 등록한다.

**드러나는 행위**

- 개인 전시 등록은 **로그인한 사용자만** 가능하다.
- 등록된 CUSTOM 전시는 **등록자 본인에게만** 노출된다.
- venueId를 지정하면 전시관에서 **장소명을, (region 미지정 시) 지역을 파생**한다. 없는 venueId면 404.
- 전시 형태가 개인전(SOLO)이면 **작가명이 필수**다.
- 장르 키워드는 등록 시점에 **분류기(랜덤/AI)가 자동 부여**한다. 분류기는 실패해도 예외를 던지지 않아 등록 흐름을 깨지 않는다.
- 전시관 자동완성은 keyword가 공백/미입력이면 **빈 목록**(에러 아님).

---

### 2.5 회원 — 관심 전시(북마크)

> 사용자는 마음에 드는 전시를 관심 등록해 두고 모아 본다.

**시나리오**

1. 사용자가 전시를 관심 등록한다.
2. 사용자가 관심 등록을 해제한다.
3. 사용자가 자신의 관심 전시 목록을 조회한다.

**드러나는 행위**

- 관심 등록/해제는 **로그인한 사용자만** 가능하며 **멱등**이다 — 이미 등록된 전시를 다시 등록해도, 이미 해제된 전시를 다시 해제해도 성공한다.
- 없는 전시에 대한 등록/해제는 404.
- (user, exhibition) 한 쌍당 **한 행** — 해제는 soft-delete, 재등록은 같은 행 복원.
- 목록 정렬: 등록 최신순(latest, 기본), 종료 임박순(ending, 종료일 없는 전시는 뒤로). 커서 페이지네이션.

---

### 2.6 회원 — 기록

> 사용자는 다녀온 전시의 감상을 감정·미디어와 함께 기록하고, 아카이브에서 다시 본다.

**시나리오**

1. 사용자가 전시를 선택해 감상(직접 작성 또는 AI 도움)·감정·관람일·사진/영상을 기록한다.
2. 사용자가 아카이브에서 자신의 기록 목록을 조회한다. 검색어·감정·전시·북마크 여부·작성 방식·관람일 범위로 필터링한다.
3. 사용자가 자신이 다녀온 전시 목록을 조회한다.
4. 사용자가 기록 상세를 조회하고, 수정하고, 삭제한다.
5. 사용자가 기록을 북마크하거나 해제한다.

**드러나는 행위**

- 기록은 **로그인한 사용자만**(게스트 포함) 작성할 수 있고, **본인 것만** 조회·수정·삭제할 수 있다.
- 기록 생성 시점의 전시 정보(제목·유형·포스터·장소·지역·카테고리·기간)가 **스냅샷으로 박제**된다 — 이후 전시가 변경·삭제되어도 기록은 영향받지 않는다.
- 감정은 **1개 이상 필수**. 관람일은 미래 불가, 미입력 시 오늘(KST).
- 미디어는 **최대 5개**, 사진 10MB·영상 100MB 이하, 정렬 순서(sortOrder) 중복 불가.
- 수정은 감상·감정·미디어의 **전체 교체** 방식.
- 기록 북마크/해제는 멱등이다.
- 목록은 **오프셋 페이지네이션**(기본 20건).

---

### 2.7 회원 — AI 감상문 ('질문으로 작성')

> 사용자는 AI가 던지는 질문에 답하고, AI가 답변을 감상문으로 다듬어 준다.

**시나리오**

1. 사용자가 전시를 지정하면 AI가 그 전시 맥락에 맞는 질문 3개를 생성한다. "다른 질문 보기"는 같은 호출의 반복이다.
2. 사용자가 질문에 답하면 AI가 답변을 1인칭 감상문(300자 이내)으로 다듬는다. "다시 다듬기"도 같은 호출의 반복이다.
3. 사용자가 다듬어진 감상문을 확인·수정한 뒤 기록 작성(2.6)으로 저장한다.

**드러나는 행위**

- AI 감상문 API는 **저장하지 않는다** — 최종 저장은 기록 생성 API가 담당한다.
- 사용자당 **최소 호출 간격(쿨다운)** 을 강제한다. 간격 내 재호출은 429(AI_RATE_LIMITED).
- 전시 접근 권한(타인의 CUSTOM 전시 금지)을 검증한 뒤 전시 맥락을 프롬프트로 만든다.
- 외부(공공데이터) 텍스트는 **참고 자료로만** 취급하는 가드 문구를 프롬프트에 넣고, 설명은 500자로 잘라 프롬프트 인젝션·토큰 폭주를 완화한다.
- AI 호출은 전용 스레드풀에서 **비동기 실행**해 서블릿 워커 스레드를 빨리 반환한다.

---

### 2.8 회원 — 리마인드(회고)

> 사용자는 7일 이상 지난 기록을 다시 꺼내어, 지금의 감정과 소감을 남긴다.

**시나리오**

1. 사용자가 오늘의 소환 대상(작성 후 7일 이상 지났고 아직 회고하지 않은 내 기록) 1건을 확인한다.
2. 사용자가 지금의 감정과 소감(오늘의 여운)을 남겨 리마인드를 저장한다. AI가 그때/지금의 감정 변화를 요약해 준다.
3. 사용자가 아카이브에서 리마인드 목록을 조회한다.
4. 사용자가 리마인드 상세(그때의 감상 vs 지금의 소감, AI 요약)를 조회한다.

**드러나는 행위**

- 소환 대상이 없으면 **null을 반환**한다(에러 아님).
- 소감(reflection)은 **필수**. 감정 코드는 공백 제거·중복 제거 후 저장한다.
- AI 감정 변화 요약은 **best-effort** — 미설정(SKIPPED)·쿨다운(SKIPPED)·실패(FAILED) 어떤 경우에도 **저장 자체는 성공**한다. AI 호출은 DB 트랜잭션 밖에서 수행한다.
- 저장 시점에 원본 기록의 전시 카드 정보(전시·포스터·장소·관람일)를 **스냅샷으로 복사**한다 — 원본 기록이 삭제돼도 리마인드 카드는 유지된다.
- 상세의 "그때"(원본 기록 내용·감정)는 **라이브 조회** — 원본이 삭제됐으면 null/빈 목록.
- 한 기록에 대해 시간차로 **여러 번** 리마인드할 수 있다(Record 1 : N Remind).
- 리마인드는 본인 것만 접근할 수 있다. 타인 접근은 403.

---

### 2.9 회원 — 알림

> 사용자는 홈 종 아이콘에서 알림 목록을 확인하고 읽음 처리한다.

**시나리오**

1. 사용자가 자신의 알림 목록을 최신순으로 조회한다.
2. 사용자가 알림을 읽음 처리한다.

**드러나는 행위**

- 알림은 **본인 것만** 조회·읽음 처리할 수 있다. 타인 알림 접근은 **404**(존재를 노출하지 않음).
- 읽음 처리는 **멱등** — 이미 읽은 알림을 다시 읽어도 결과는 동일.
- 알림 종류: REMIND(리마인드, targetId=remindId로 이동)·NOTICE(공지, 이동 대상 없음 targetId=null).
- 목록은 커서 페이지네이션(최신순).

### 2.10 시스템 — 전시 카탈로그 동기화·보강

> 시스템은 외부 공공데이터 전시 API를 주기적으로 수집해 DB에 적재하고, 장르·상세 정보를 채운다.

**시나리오**

1. 부팅 직후 전시 카탈로그를 1회 동기화한다(cold start 방지).
2. 매일 자정 목록을 재동기화하고(신규만 적재), 상세(가격 등)를 함께 백필한다.
3. 장르 분류는 별도 주기 없이 **동기화 직후에만** 수행된다 — 부팅 동기화 후 데몬 스레드로 1회(기동 비차단) + 매일 자정 동기화 직후. 대상이 미분류 행이라 방금 적재된 신규분만 분류한다. 장르 분류기는 운영은 AI(gemini), 개발·로컬은 랜덤이다.

**드러나는 행위**

- 수집 전시는 **externalId 기준 신규만 insert** — 이미 있으면 스킵, 없으면 신규 적재.
- **기존 행은 재적재로 갱신하지 않는다** — 장르(AI 분류)·상세 등 보강으로 채운 값을 정기 동기화가 덮어쓰지 않게.
- 기간이 비정상(종료일 < 시작일)인 원천 레코드는 **건너뛰고 계속** 적재한다(배치 전체 중단 방지).
- 장르 분류는 동기화 직후 신규 적재분(미분류 전시)을 **배치당 1회 호출**로 일괄 분류한다. 운영은 AI(gemini) 분류로 실패(429 등) 시 랜덤 폴백되며, 개발·로컬은 처음부터 랜덤 분류기를 쓴다. 실행 자체가 실패하면 다음 동기화에서 재시도된다.
- 상세 백필은 행 단위 짧은 트랜잭션으로 수행하고, 원천에 상세가 없으면 확인 완료로 표기해 재조회를 막는다.
- 보강 대상은 "미처리 행"이라 **멱등** — 반복 실행돼도 신규 행만 처리한다.
- 인증키 미설정이면 외부 호출 없이 0건으로 끝난다. 데모 시드는 별도 플래그로만 켠다.

## 4. 예외 및 정책

### 4.1 삭제 정책

- 삭제는 **논리적 삭제(Soft Delete)** 를 원칙으로 한다(users, exhibitions, records, reminds, exhibition_bookmarks, notifications, venues).
- 삭제된 데이터는 조회되지 않는다(조회는 deleted_at IS NULL 행만).
- **애그리거트 내부 자식**(record_emotions, record_keywords, record_media, remind_emotions)은 deleted_at 없이 부모 수정 시 **전체 교체(물리 삭제)** 된다.
- 관심 전시 해제는 soft-delete지만 **재등록 시 같은 행을 복원**한다(UNIQUE 제약과 멱등성 공존).
- 원본 기록이 삭제되어도 **리마인드의 스냅샷 카드는 유지**된다. 기록의 전시 스냅샷도 마찬가지다.

### 4.2 인증 정책

- 인증은 자체 JWT access 토큰으로 한다. **HttpOnly 쿠키 또는 Bearer 헤더** 어느 쪽이든 검증한다.
- 인증 유저는 컨트롤러 파라미터(`@Authentication LoginUser`)로 주입된다. 인증 실패는 401.
- 전시 목록·상세는 **선택 인증**(`@OptionalAuthentication`) — 비로그인도 접근 가능하되 로그인 시 개인화가 반영된다.

### 4.3 존재하지 않거나 접근 불가한 자원

- 존재하지 않거나 삭제된 자원의 조회/수정 요청은 **404 NOT_FOUND**로 응답한다.
- 타인의 자원 접근은 도메인에 따라 다르다 — **기록·리마인드·CUSTOM 전시는 403 FORBIDDEN**(소유자 아님을 명시), **알림은 404**(존재 여부를 노출하지 않음).
- 손상되었거나 정렬 기준이 불일치하는 커서는 400 INVALID_CURSOR.

### 4.4 페이징 정책

- **커서(키셋) 페이지네이션** — 전시 목록, 관심 전시 목록, 알림 목록. 기본 20건, 최대 50건. 응답에 nextCursor·hasNext·totalCount 포함.
- **오프셋 페이지네이션**(Pageable) — 기록 목록, 리마인드 목록. 기본 20건.
- 커서는 정렬 판별자(sort)를 함께 인코딩하며, 현재 정렬과 불일치하면 거부한다.

### 4.5 AI 정책

- LLM provider는 포트(AiChatClient)로 추상화한다 — Claude/Gemini 구현을 설정으로 전환한다.
- 장르 분류기(GenreClassifier)도 랜덤/AI(Gemini) 구현을 설정으로 선택한다. **분류 결과는 장르 키워드 마스터(10종) 안의 값**으로 강제한다.
- 장르 AI(app.ai.gemini, 기본 gemini-2.5-flash)와 감상문 AI(app.ai, 기본 gemini-2.5-flash-lite)는 **기본 모델을 다르게** 둔다 — Gemini 무료 한도는 모델별이라 호출 버킷이 분리된다.
- 사용자 대면 AI(질문·감상문)는 **동기 + 쿨다운**, 부가 AI(리마인드 요약)는 **best-effort**(실패해도 본 작업 성공), 배치 AI(장르 분류)는 **배치당 1콜 + 폴백**.
- AI 미설정 환경에서도 서비스의 핵심 흐름(기록·리마인드 저장)은 동작한다.

| 알림 생성(발송) 배치 | 알림 조회·읽음 처리만 구현. 리마인드 도래 시 알림을 만들어 넣는 생산자(스케줄러)는 향후 단계 |
| 어드민 기능 | 공지 등록·전시 운영 관리 등 어드민 API 없음. 배너도 운영자 지정 대신 조회수 상위 자동 선정 |
| 미디어 파일 업로드 | 기록 미디어는 URL·크기만 받는다. 실제 파일 업로드(스토리지)는 별도 채널 |
| 기록 AI 후처리 필드 | 요약·대표감정·카드문구는 스키마만 유지하고 항상 null(아카이브 카드 정식화 시 재도입) |
| 기록 키워드 | 키워드 개념 폐지 — 테이블은 남아 있으나 수정 시 정리되는 레거시 |
| 소셜 계정 추가 연결/해제 | 로그인 시 자동 연결만 존재. 계정 관리 화면용 연결 관리 API 없음 |
| 검색 엔진·추천 | 검색은 LIKE 부분 일치. 취향 기반 추천은 감정 키워드 노출까지만 |
