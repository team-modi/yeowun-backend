> 기록 스냅샷(V8) 반영. 백엔드 마지막 태스크로서 구현 완료 상태 기준으로 재정리.

## 4.1 무엇을 만들었나 (한눈에)

| METHOD | URI | 인증 | 설명 | 상태 |
| --- | --- | --- | --- | --- |
| GET | `/api/v1/exhibitions` | 선택 | 목록/탐색/검색(keyword·date·region·category·page/size, sort=latest\|ending\|popular) | ✅ |
| GET | `/api/v1/exhibitions/{exhibitionId}` | 선택 | 상세(CATALOG 공개 / CUSTOM 본인만) — 최초 진입 시 상세(detail2) 지연수집 + 조회수 +1 | ✅ |
| POST | `/api/v1/exhibitions/custom` | 필수 | 개인 전시 등록 | ✅ |
| GET | `/api/v1/records/exhibitions/visited` | 필수 | 내가 다녀온 전시 목록(기록 작성 시점 전시 스냅샷 노출) — Record 도메인, 여기선 스냅샷 독립성만 요약 | ✅ |

- **CATALOG**(공용 전시): 공공데이터 API(한눈에보는문화정보, data.go.kr **15138937**, `apis.data.go.kr/B553457/cultureinfo`)의
  `realm2`(목록, 분야코드 D000=전시) 응답을 **배치로 수집→DB upsert**. 응답 포맷은 XML(`CultureApiResponse` + `XmlMapper`).
- **CUSTOM**(개인 전시): 사용자가 등록, 등록자 본인에게만 노출.
- **선택 인증**: 목록·상세는 비로그인 가능. 토큰이 있으면 본인 CUSTOM까지 함께 노출(`@OptionalAuthentication`).
- **동기화**: 부팅 시 1회(`ExhibitionCatalogBootSync`, cold start 방지) + 매시 정각(`ExhibitionSyncScheduler`, `@Scheduled(cron="0 0 * * * *")`).
  인증키 미설정이면 외부 호출 없이 0건으로 안전하게 스킵.
- **상세 지연수집**: 목록(`realm2`)엔 없는 주소·이미지·전화·가격 등은 상세 화면 최초 진입 시 1회 `detail2`를 호출해 캐시
  (`Exhibition.applyDetail`, `detailSyncedAt`로 재수집 여부 판단). 이후 조회는 DB만 본다.
- **인기순 정렬**: 외부 API가 아닌 **우리 앱 내 조회수**(`ourViewCount`, 상세 조회마다 +1)를 기준으로 `sort=popular` 정렬.
- **기록 스냅샷**: 기록(Record) 작성 시점의 전시 정보(title·type·posterUrl·place·region·category·기간)를 `records` 테이블에
  그대로 복제해 저장(`ExhibitionSnapshot`, V8). 이후 전시 원본이 재동기화·수정·삭제돼도 기록의 스냅샷은 바뀌지 않는다
  — `RecordV1ControllerTest`의 e2e 케이스로 검증(4.5 참고).
- 검증: `./gradlew build`(전 스위트 그린), `ExhibitionIntegrationTest`(정렬·지역필터·상세 지연수집 포함) +
  `RecordV1ControllerTest`(기록 스냅샷 독립성 e2e), 실 API 인증키로 `CultureApiLiveE2ETest` 그린(4.4 질문 1 해결).

## 4.2 아키텍처 결정 (왜 이렇게 했나)

### (A) CATALOG는 "수집-적재(동기화)" 방식 — 실시간 외부 호출 아님 (최종 확정)
공공데이터 리뷰.pdf가 지적한 **호출 한도·응답 결측·안정 식별자(seq)** 특성상, 요청마다 외부 API를 때리는 대신
`external_id`(원천 seq) 기준으로 DB에 upsert하고 **조회/등록은 DB만** 본다. 최종 구현은 아래처럼 세 층으로 나뉜다.

1. **목록 배치 동기화(realm2)** — `ExhibitionCatalogBootSync`(부팅 시 1회, cold start 방지) +
   `ExhibitionSyncScheduler`(`@Scheduled(cron="0 0 * * * *")`, 매시 정각)가 `ExhibitionFacade.syncCatalog()`를 호출한다.
   `CultureExhibitionClient`가 `realm2`를 `numOfRows`(기본 100) × `maxPages`(기본 5) 페이지네이션으로 순회 수집하고,
   `externalId` 기준 upsert(있으면 `refreshCatalog`, 없으면 `createCatalog`)한다. 둘 다 외부 실패를 `RuntimeException` 캐치로
   삼켜 다음 주기에 재시도하고, 인증키 미설정이면 `PublicDataProperties.isConfigured()==false`로 외부 호출 자체를 스킵한다.
2. **상세 지연수집(detail2)** — `realm2`엔 없는 가격·설명·주소·이미지·전화·원문 상세링크는 상세 화면 **최초 진입 시 1회만**
   `detail2`(`?seq=`)를 호출해 캐시한다(`ExhibitionFacade.getDetail` → `Exhibition.applyDetail`, `detailSyncedAt`으로 재수집
   여부 판단). 외부 실패 시 base 필드만 응답하고 `detailSyncedAt`이 null로 남아 다음 조회에서 재시도된다.
3. **응답 포맷은 XML** — `realm2`/`detail2` 모두 XML 응답이라 WebClient의 JSON 코덱 대신 문자열로 받아 `XmlMapper`로
   직접 파싱한다(`CultureApiResponse`). `header.resultCode!="00"`이면 `EXTERNAL_API_UNAVAILABLE`로 변환.

- 장점: 통합 페이지네이션·필터·정렬을 한 테이블에서 일관되게 처리, 외부 장애가 조회를 막지 않음, 상세 지연수집으로
  불필요한 API 호출(목록에 있는 항목 전부의 상세를 미리 당기지 않음) 최소화.
- 트레이드오프: 데이터 신선도는 최대 1시간 지연(스케줄러 주기) — 4.4 오픈 질문 6(해결됨) 참고.
- **인기순(`sort=popular`)**: 외부 API 조회수가 아니라 우리 앱 상세 진입 횟수(`Exhibition.ourViewCount`, `increaseView()`)로
  집계한다 — 원천이 조회수를 제공하지 않고, 우리 서비스 내 관심도를 반영하는 게 목적에 더 맞다는 판단.

### (B) CATALOG·CUSTOM을 한 테이블(`exhibitions`)로
03_전시.md 결정(개인전을 독립 엔티티로)을 따름. `type` 컬럼으로 구분, `owner_id`로 CUSTOM 노출 제어.
통합기획서 안건11(개인전을 Record 내장으로 둘지)은 **미해결** — 4.4 오픈 질문 4.

### (C) 동적 필터 = Spring Data JPA Specification
프로젝트에 QueryDSL 의존이 없어(빌드 확인) `JpaSpecificationExecutor`로 조합 쿼리를 구성.
soft-delete(`deleted_at is null`)·CUSTOM 노출·keyword·기간·region·category를 하나의 Specification으로 묶었다.

### (D) 계층/의존은 기존 컨벤션 준수
`Controller → Facade → Entity ← Infra`. 상태 변경은 `Exhibition` 메서드 안에서만(팩토리·`refreshCatalog`).
외부 HTTP는 도메인 포트(`ExhibitionCatalogClient`)로 감추고 infra(`CultureExhibitionClient`)가 DIP로 구현.
에러는 `ExhibitionErrorCode`(NOT_FOUND·EXTERNAL_API_UNAVAILABLE) + 공통 `ErrorType`(INVALID_INPUT·FORBIDDEN·UNAUTHORIZED) 재사용.

### (E) 성공 코드 200 (명세의 201 대신) — **컨벤션 우선**
03_전시.md는 등록 성공에 `201`을 적었지만, `.claude/CLAUDE.md`는 "성공 전부 200(201 ❌)"을 **의도된 선택**으로 명시.
프로젝트 컨벤션을 우선해 등록도 `200 + {exhibitionId, type}`로 응답. → 확정 필요(4.4 질문 2).

## 4.4 ⚠️ 미해결 · 확인 필요 (질문 대신 여기 남김)

> 임의로 현명하게 결정한 뒤, 확정이 필요한 것들을 모았다. 우선순위 순.

1. **[해결됨] 공공데이터 인증키·엔드포인트 검증 완료**
   - data.go.kr **15138937 일반 인증키**(64자, `.env`의 `CULTURE_API_KEY`)로 실호출 검증 완료.
     base-url은 최종 `https://apis.data.go.kr/B553457/cultureinfo`, 목록은 `realm2`(분야코드 D000=전시), 상세는 `detail2`.
     (이전 기록의 `http://api.kcisa.kr/...` 추정값은 폐기 — 실제 소스와 다른 API였다.)
   - 응답 포맷은 JSON이 아니라 **XML**이었다 — `CultureApiResponse` + `XmlMapper`로 재작성(`comMsgHeader/msgBody/perforList`
     추정 스키마도 폐기, 실제는 `header/body/items/item`).
   - `CULTURE_API_KEY=발급키 ./gradlew test --tests '*CultureApiLiveE2ETest'` 그린(2026-07-02, Task 14 재검증) —
     실 API가 비어 있지 않은 수집 데이터를 반환하고 전 건 `externalId`·`title`을 갖춤을 확인.

2. **[해결됨] 성공 코드 200 vs 201**: CLAUDE.md 컨벤션(성공 전부 200, 201/PATCH 금지)이 프로젝트 전역 확정 규칙 —
   등록 포함 전부 200 유지로 최종 확정.

3. **region/category enum 확정 + 외부 매핑표**: 03_전시.md에서 `⚠가정`. 외부 `area`→region, `realmName`→category는
   여전히 **휴리스틱**(`fromAreaText`/`fromRealmName`)이며, 원문 값은 `area_text`/`realm_name` 컬럼(V7)에 그대로 보존해
   재매핑 여지를 남겼다. 팀 확정 코드셋과 매핑 규칙(특히 realmName 세부값)은 여전히 미해결.

4. **개인전 모델링(통합기획서 안건11)**: 독립 엔티티(`type=CUSTOM`)로 구현. Record 내장(`customExhibition`) 안과의 트레이드오프
   미해결(담당 협의). 방향 확정 시 마이그레이션/모델 조정 여지.

5. **artists·keywords 원천 미제공**: 상세 응답의 `artists`/`keywords`는 소스 A(한눈에보는문화정보)에 없어 **항상 빈 배열**.
   보강 소스(리뷰 B: 문체부 12개 기관 15105037의 GENRE/DESCRIPTION 등) 연동 또는 필드 제거 결정 필요. (미해결)

6. **[해결됨] 동기화 트리거·주기**: 부팅 시 1회(`ExhibitionCatalogBootSync`, cold start 방지) + 매시 정각
   (`ExhibitionSyncScheduler`, `@Scheduled(cron="0 0 * * * *")`) 재동기화로 확정. 관리자 수동 트리거 API는 여전히 없음
   (필요해지면 별도 엔드포인트로 추가).

7. **keyword 작가명 검색 불가**: 명세는 "작가명 부분 일치" 포함이나 원천에 작가 필드가 없어 **title·place만** 검색한다.
   (5번과 연동, 미해결)

8. **[해결됨] 기본 정렬**: `sort=latest|ending|popular`(기본 latest=시작일 최신순)로 확정 구현.
   `ending`=종료일 임박순, `popular`=우리 앱 조회수(`ourViewCount`)순. 화이트리스트 밖 값은 default(latest)로 안전 처리
   (`ExhibitionFacade.toSort`) — 500 이슈는 해소됨.

9. **좌표·원문링크·제공기관 노출(반영됨)**: 상세 응답(`DetailResponse`)에 `gpsX`·`gpsY`·`detailUrl`·`serviceName`에 더해
   Task 14 재작업으로 `address`(장소 상세주소)·`imgUrl`·`phone`·`viewCount`까지 추가해 원천의 실제 응답 데이터를 전부
   내려준다(명세 3.3.2보다 확장). 목록(`ListItemResponse`)은 카드용 최소 필드만 유지.
   → 명세 응답 스키마를 이 확장으로 확정할지 검토는 여전히 필요.

10. **멱등성**: 명세의 `RULE: 멱등성`(등록)은 미구현(동일 제목 중복 허용만 반영). 멱등키가 필요하면 별도 설계. (미해결)

11. **ongoing(진행 중) 경계**: 시작/종료일이 결측인 CATALOG는 각각 "이미 시작"/"아직 진행"으로 **관대하게** 포함한다.
    엄격히 둘 다 있어야 진행 중으로 볼지 정책 확정 필요. (미해결)

12. **[신규, Phase 2] 기록 스냅샷의 전시 원본 참조 유지 여부**: 기록은 작성 시점 전시 정보를 통째로 복제(`ExhibitionSnapshot`)해
    원본과 독립적으로 보존한다(의도된 설계 — 전시가 삭제·수정돼도 사용자의 관람 기록은 훼손되지 않아야 함). 다만 원본
    `exhibitionId`도 함께 들고 있어 원본이 살아있다면 "지금 이 전시 보러가기" 같은 딥링크는 여전히 가능하다. 원본이
    삭제된 경우의 딥링크 처리(404 UX)는 프론트 협의 필요(미해결).

> 제목이 여전히 최초 작성 시점 값임을 검증한다.
