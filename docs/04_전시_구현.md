# 4. 전시 도메인 구현 정리

> 03_전시.md 명세 + 공공데이터 리뷰.pdf를 바탕으로 전시 기능을 구현한 결과 정리 문서.
> 실제 코드가 진실의 원천이며, 이 문서는 **의사결정 근거**와 **미해결(질문할) 항목**을 남기기 위한 것이다.
> 작성 시점: 2026-07-01. 개발 방식: ai-driven-tdd-development-skill(Red→Green→Refactor), 통합테스트 중심.

## 4.1 무엇을 만들었나 (한눈에)

| METHOD | URI | 인증 | 설명 | 상태 |
| --- | --- | --- | --- | --- |
| GET | `/api/v1/exhibitions` | 선택 | 목록/탐색/검색(keyword·date·region·category·page/size/sort) | ✅ |
| GET | `/api/v1/exhibitions/{exhibitionId}` | 선택 | 상세(CATALOG 공개 / CUSTOM 본인만) | ✅ |

- **CUSTOM**(개인 전시): 사용자가 등록, 등록자 본인에게만 노출.
- **선택 인증**: 목록·상세는 비로그인 가능. 토큰이 있으면 본인 CUSTOM까지 함께 노출(`@OptionalAuthentication`).
- 검증: `./gradlew build` (전 스위트 그린), 통합테스트 14케이스(`ExhibitionIntegrationTest`), 실 서버 부팅 후 3개 엔드포인트·데모 페이지 실동작 확인.

## 4.2 아키텍처 결정 (왜 이렇게 했나)

### (A) CATALOG는 "수집-적재(동기화)" 방식 — 실시간 외부 호출 아님
공공데이터 리뷰.pdf가 지적한 **호출 한도·응답 결측·안정 식별자(seq)** 특성상, 요청마다 외부 API를 때리는 대신
`external_id`(원천 seq) 기준으로 DB에 upsert하고 **조회/등록은 DB만** 본다.
- 장점: 통합 페이지네이션·필터·정렬을 한 테이블에서 일관되게 처리, 외부 장애가 조회를 막지 않음.
- 트레이드오프: 데이터 신선도는 동기화 주기에 의존(→ 4.4 오픈 질문 6).

### (B) CATALOG·CUSTOM을 한 테이블(`exhibitions`)로
03_전시.md 결정(개인전을 독립 엔티티로)을 따름. `type` 컬럼으로 구분, `owner_id`로 CUSTOM 노출 제어.
통합기획서 안건11(개인전을 Record 내장으로 둘지)은 **미해결** — 4.4 오픈 질문 4.

### (C) 동적 필터 = Spring Data JPA Specification
프로젝트에 QueryDSL 의존이 없어(빌드 확인) `JpaSpecificationExecutor`로 조합 쿼리를 구성.
soft-delete(`deleted_at is null`)·CUSTOM 노출·keyword·기간·region·category를 하나의 Specification으로 묶었다.

### (D) 계층/의존은 기존 컨벤션 준수
`Controller → Facade → Entity ← Infra`. 상태 변경은 `Exhibition` 메서드 안에서만(팩토리·`refreshCatalog`).
외부 HTTP는 도메인 포트(`ExhibitionCatalogClient`)로 감추고 infra(`CultureExhibitionClient`)가 DIP로 구현.
에러는 `ExhibitionErrorCode`(NOT_FOUND·EXTERNAL_API_UNAVAILABLE) + 공통 `ErrorType`(INVALID_INPUT·FORBIDDEN·UNAUTHORIZED) 재사용.

### (E) 성공 코드 200 (명세의 201 대신) — **컨벤션 우선**
03_전시.md는 등록 성공에 `201`을 적었지만, `.claude/CLAUDE.md`는 "성공 전부 200(201 ❌)"을 **의도된 선택**으로 명시.
프로젝트 컨벤션을 우선해 등록도 `200 + {exhibitionId, type}`로 응답. → 확정 필요(4.4 질문 2).

## 4.4 ⚠️ 미해결 · 확인 필요 (질문 대신 여기 남김)

> 임의로 현명하게 결정한 뒤, 확정이 필요한 것들을 모았다. 우선순위 순.

1. **[치명적] 공공데이터 인증키·엔드포인트 미검증**
   - `CULTURE_API_KEY`를 받지 못해 **실제 API 응답으로 검증하지 못했다.** base-url 기본값
     `http://api.kcisa.kr/openapi/API_CCA_145/request`은 리뷰 PDF의 응답 스키마(comMsgHeader/msgBody/perforList)로 **추정**한 값.
   - 할 일: data.go.kr에서 **Decoding 인증키** 발급 → `CULTURE_API_KEY`(+필요시 `CULTURE_API_BASE_URL`) 설정 →
     `CULTURE_API_KEY=키 ./gradlew test --tests '*CultureApiLiveE2ETest'`로 실호출 검증.
   - 실제 필드명이 PDF와 다르면 `CultureApiResponse`(필드) / `CultureExhibitionClient`(매핑)만 조정하면 됨.
   - data.go.kr 응답은 단건일 때 배열 대신 객체로 오는 특성이 있어 `ACCEPT_SINGLE_VALUE_AS_ARRAY`를 켜 뒀다(추정 대응).

2. **성공 코드 200 vs 201**: 03_전시.md는 등록 `201`, CLAUDE.md 컨벤션은 `200`. **컨벤션 우선해 200**으로 구현. 최종 확정 필요.

3. **region/category enum 확정 + 외부 매핑표**: 03_전시.md에서 `⚠가정`. 외부 `area`→region, `realmName`→category는
   현재 **휴리스틱**(`fromAreaText`/`fromRealmName`). 팀 확정 코드셋과 매핑 규칙(특히 realmName 세부값)이 필요.

4. **개인전 모델링(통합기획서 안건11)**: 독립 엔티티(`type=CUSTOM`)로 구현. Record 내장(`customExhibition`) 안과의 트레이드오프
   미해결(담당 협의). 방향 확정 시 마이그레이션/모델 조정 여지.

5. **artists·keywords 원천 미제공**: 상세 응답의 `artists`/`keywords`는 소스 A(한눈에보는문화정보)에 없어 **항상 빈 배열**.
   보강 소스(리뷰 B: 문체부 12개 기관 15105037의 GENRE/DESCRIPTION 등) 연동 또는 필드 제거 결정 필요.

6. **동기화 트리거·주기 미정**: 현재 `syncCatalog()`는 데모 시드(부팅 시)에서만 호출. 운영은 **스케줄러(야간 배치)** 또는
   **관리자 API**가 필요 — 미구현. 주기/트리거/실패 재시도 정책 결정 필요.

7. **keyword 작가명 검색 불가**: 명세는 "작가명 부분 일치" 포함이나 원천에 작가 필드가 없어 **title·place만** 검색한다.
   (5번과 연동)

8. **기본 정렬**: `startDate,DESC`로 가정(명세 `⚠가정`). 확정 필요. (잘못된 sort 필드는 현재 500 — 필요 시 화이트리스트 방어 추가)

9. **좌표·원문링크·제공기관 노출(반영됨)**: 상세 응답(`DetailResponse`)에 `gpsX`·`gpsY`·`detailUrl`·`serviceName`를 추가해
   원천의 실제 응답 데이터를 전부 내려주도록 했다(명세 3.3.2보다 확장). 목록(`ListItemResponse`)은 카드용 최소 필드만 유지.
   → 명세 응답 스키마를 이 확장으로 확정할지 검토 필요.

10. **멱등성**: 명세의 `RULE: 멱등성`(등록)은 미구현(동일 제목 중복 허용만 반영). 멱등키가 필요하면 별도 설계.

11. **ongoing(진행 중) 경계**: 시작/종료일이 결측인 CATALOG는 각각 "이미 시작"/"아직 진행"으로 **관대하게** 포함한다.
    엄격히 둘 다 있어야 진행 중으로 볼지 정책 확정 필요.
