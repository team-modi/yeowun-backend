# 1. 인증/인가 도메인

> 공통 규칙(응답 포맷, 에러 코드, 페이지네이션, 미디어, 멱등성)은 [00_개요_및_공통규칙.md](./00_개요_및_공통규칙.md) 참조. User 엔티티 자체(닉네임 RULE, 프로필 필드)는 [02_유저.md](./02_유저.md) 참조.

## 1.0 개요
- **한 줄 설명**: 카카오·네이버 소셜 로그인으로 인증하고 서비스 Access/Refresh 토큰을 발급한다. 모든 API의 인증 헤더 규약과 리소스 접근 권한(인가) 정책을 정의한다.
- **용어 정의**
  - `provider`: 소셜 공급자. `KAKAO` | `NAVER` (Must). `GOOGLE`은 통합기획서 안건3에서 P1 후보로 언급되었으나 본 명세 1차 범위에는 포함하지 않는다 — 추후 enum에 추가만 하면 되는 구조로 설계.
  - `Access Token`: 소셜 로그인 성공 후 발급되는 우리 서비스 자체 JWT. 모든 인증 필요 API에서 사용. 수명이 짧다(`⚠가정` 2시간).
  - `Refresh Token`: Access Token 재발급에 쓰는 별도 토큰. 수명이 길다(`⚠가정` 14일). 통합기획서 P3 개발기능표에 "access/refresh token" 발급이 명시되어 있어, **이전 버전에서 범위 밖으로 가정했던 Refresh Token을 1차 범위로 편입**했다(부록 B 참조).
- **범위 밖**: 자체 회원가입(ID/PW), 비밀번호, 이메일 인증, 회원 탈퇴.
- **결정 사항**: 회원(User) 엔티티의 생성·필드·닉네임 규칙은 이 도메인이 아니라 [02_유저.md](./02_유저.md)가 소유한다. 이 도메인은 "누가 인증되었고 무엇에 접근 가능한가"만 다루며, 소셜 로그인 처리 중 신규 회원 생성 단계에서 유저 도메인의 `RULE: 닉네임`을 그대로 따른다.
- **redirectUrl 복귀**: 통합기획서 P3는 "로그인 후 원래 목적지로 복귀(redirectUrl)"를 요구하지만, 본 명세의 OAuth 흐름은 **프론트엔드가 인가 코드를 받아 백엔드로 POST하는 방식**(서버 리다이렉트 방식 아님)이라 redirectUrl 보존·복귀는 **프론트엔드 라우팅 책임**이다. 백엔드는 토큰 발급까지만 담당하며 별도 API 처리가 필요 없다. `⚠가정`(서버 리다이렉트형 OAuth로 바뀌면 `state` 파라미터 처리가 필요해진다.)

## 1.1 인증 방식 `⚠가정` (문서에 OAuth 핸드셰이크 미명시 → 표준 방식으로 확정)
- 소셜 로그인(카카오·네이버)으로만 인증한다. 자체 ID/PW 없음.
- **흐름**: 프론트가 소셜 인가 → `authorization code` 획득 → 백엔드 `POST /auth/social-login`으로 전달 → 백엔드가 소셜 토큰 교환 + 프로필 조회 → 우리 회원 upsert(`02_유저.md`) → **서비스 Access Token + Refresh Token 발급**.
- 인증이 필요한 요청은 헤더로 Access Token을 싣는다.
  - `Authorization: Bearer {accessToken}`
- Access Token 만료(기본 `⚠가정` 2시간) 시 → `401 / TOKEN_EXPIRED`. 클라이언트는 **Refresh Token으로 재발급**(`POST /auth/refresh`) 후 원래 요청을 재시도하고, Refresh Token까지 만료/무효면 재로그인한다. (유저플로우 로그인_2_A의 "세션 만료 안내 → 소셜 로그인 → 저장 재시도" 경로는 Refresh Token도 만료된 최악의 경우에 해당)
- 토큰 누락/위조/무효 → `401 / UNAUTHORIZED`.

**RULE: Refresh Token** `⚠가정`
- 수명: 14일. Access Token(2시간)보다 길게 가져가 재로그인 빈도를 낮춘다.
- 저장 위치: 클라이언트(예: secure storage). 서버는 발급 이력만 보관(검증·무효화용).
- 재발급 시 회전(rotation) 여부: `⚠가정` **회전하지 않음**(1차 단순화). 탈취 대응이 중요해지면 회전 방식으로 전환.
- 로그아웃 시 해당 Refresh Token 즉시 무효화(1.4.2 참조).

## 1.2 권한(인가) 기본 규칙
- **전시 조회/검색**은 공개(비로그인 가능).
- **기록·리마인드·프로필·북마크·미디어·개인 전시**는 모두 **본인 리소스만** 접근/조작 가능.
  - 남의 리소스 ID로 조회/수정/삭제 시도 → `403 / FORBIDDEN`.
  - 단, 리소스 존재를 숨겨야 하는 경우(타인 기록 상세 등)는 `404 / NOT_FOUND`로 응답할지 `403`으로 응답할지 **정책 결정**: 본 명세는 **소유자 불일치 시 `403 / FORBIDDEN`** 으로 통일. `⚠가정`
- 인증 필요한 API에 비로그인 접근 → `401 / UNAUTHORIZED`.

## 1.3 API 목록
| METHOD | URI | 인증 필요 | 설명 |
| --- | --- | --- | --- |
| POST | `/api/v1/auth/social-login` | X | 소셜 로그인(가입 겸용) |
| POST | `/api/v1/auth/refresh` | X (Refresh Token 필요) | Access Token 재발급 |
| POST | `/api/v1/auth/logout` | O | 로그아웃 |

## 1.4 API 상세

### 1.4.1 소셜 로그인 — `POST /api/v1/auth/social-login`
- **인증 필요**: X
- **권한**: 누구나

**Request (body)**
| 필드 | 타입 | 필수 | 제약 | 예시 |
| --- | --- | --- | --- | --- |
| provider | string(enum) | O | `KAKAO` \| `NAVER` | `"KAKAO"` |
| authorizationCode | string | O | 소셜에서 발급된 인가 코드 | `"qX1abc..."` |
| redirectUri | string | X | `⚠가정` 소셜 콘솔 등록값과 일치 | `"https://yeoun.example/oauth/kakao"` |

**처리 로직**
1. `provider`별 토큰 엔드포인트에 `authorizationCode` 교환 요청 → 소셜 access token 획득.
2. 소셜 프로필 조회(소셜 고유 ID, 닉네임, 프로필 이미지, `⚠가정` 이메일).
3. `(provider, socialId)`로 회원 조회.
   - 없으면 신규 생성(닉네임은 [02_유저.md](./02_유저.md)의 `RULE: 닉네임`에 따라 세팅) → `isNewUser=true`.
   - 있으면 기존 회원 → `isNewUser=false`.
4. 우리 서비스 Access Token + Refresh Token(JWT) 발급.

**Success Response**
- Status: 신규 가입 `201` / 기존 로그인 `200`
```json
{
  "meta": { "result": "SUCCESS" },
  "data": {
    "accessToken": "eyJhbGci...",
    "refreshToken": "rf_8f2c...",
    "tokenType": "Bearer",
    "expiresIn": 7200,
    "refreshExpiresIn": 1209600,
    "isNewUser": true,
    "user": { "userId": 1024, "nickname": "여운러", "profileImageUrl": "https://..." }
  }
}
```

**Error Cases**
| 상황 | HTTP Status | errorCode | 메시지(예시) |
| --- | --- | --- | --- |
| provider 미지원/오타 | 400 | `INVALID_INPUT` | 지원하지 않는 로그인 방식입니다 |
| 인가 코드 만료/무효 | 401 | `SOCIAL_AUTH_FAILED` | 소셜 인증에 실패했습니다 |
| 소셜 API 장애 | 503 | `EXTERNAL_API_UNAVAILABLE` | 잠시 후 다시 시도해 주세요 |

**예외/엣지 케이스**
- 동일 인가 코드 재사용(중복 클릭) → 소셜이 거부 시 `401 / SOCIAL_AUTH_FAILED`.
- 소셜 닉네임 미동의/없음 → 자동 닉네임 생성([02_유저.md](./02_유저.md) `RULE: 닉네임`).

**인수 조건 (Given–When–Then)**
- Given 가입 이력 없는 카카오 사용자, When 유효한 인가 코드로 로그인, Then `201` + 회원 생성 + `isNewUser=true` + 토큰 발급.
- Given 기존 회원, When 재로그인, Then `200` + `isNewUser=false`.
- Given 만료된 인가 코드, When 로그인, Then `401 / SOCIAL_AUTH_FAILED`.

### 1.4.2 토큰 재발급 — `POST /api/v1/auth/refresh`
- **인증 필요**: X (대신 Refresh Token 필요)
- **권한**: 토큰 소유자 본인

**Request (body)**
| 필드 | 타입 | 필수 | 제약 | 예시 |
| --- | --- | --- | --- | --- |
| refreshToken | string | O | 발급된 Refresh Token | `"rf_8f2c..."` |

**처리 로직**
1. Refresh Token 유효성·만료 검증.
2. 유효하면 새 Access Token 발급(`⚠가정` Refresh Token은 회전하지 않으므로 그대로 재사용).
3. 무효/만료면 재로그인 유도.

**Success Response** — `200`
```json
{
  "meta": { "result": "SUCCESS" },
  "data": { "accessToken": "eyJhbGci...", "tokenType": "Bearer", "expiresIn": 7200 }
}
```

**Error Cases**
| 상황 | HTTP Status | errorCode | 메시지(예시) |
| --- | --- | --- | --- |
| Refresh Token 누락 | 400 | `INVALID_INPUT` | refreshToken이 필요합니다 |
| Refresh Token 만료/무효 | 401 | `REFRESH_TOKEN_INVALID` | 다시 로그인해 주세요 |

**예외/엣지 케이스**
- 로그아웃으로 무효화된 Refresh Token으로 재발급 시도 → `401 / REFRESH_TOKEN_INVALID`.

**인수 조건**
- Given 유효한 Refresh Token, When 재발급 요청, Then `200` + 새 Access Token.
- Given 만료된 Refresh Token, When 재발급 요청, Then `401 / REFRESH_TOKEN_INVALID` + 클라이언트는 재로그인.

---

### 1.4.3 로그아웃 — `POST /api/v1/auth/logout`
- **인증 필요**: O
- **권한**: 본인

**Request**: body 없음. 헤더 `Authorization`만.

**처리 로직** `⚠가정`
1. 토큰 식별.
2. 해당 사용자의 Refresh Token을 즉시 무효화(폐기 목록 등록 또는 삭제).
3. (선택) Access Token도 서버 측 블랙리스트에 등록해 만료 전 무효화. 미구현 시 Access Token은 자연 만료(최대 2시간) 시까지만 유효, Refresh Token 무효화로 재발급은 막힌다.

**Success Response** — `200`, `data: null`.

**Error Cases**
| 상황 | HTTP Status | errorCode |
| --- | --- | --- |
| 토큰 누락/무효 | 401 | `UNAUTHORIZED` |

**인수 조건**
- Given 로그인 상태, When 로그아웃, Then `200` + 이후 Refresh Token으로 재발급 시도 시 `401 / REFRESH_TOKEN_INVALID`.
